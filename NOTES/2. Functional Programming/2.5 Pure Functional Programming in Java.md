# Pure Functional Programming Rules in Java

## Introduction

Pure Functional Programming follows a strict set of rules that make code:

* Predictable
* Reusable
* Testable
* Maintainable
* Easier to debug

The goal of Pure Functional Programming is to avoid unexpected behavior caused by changing state and side effects.

---

# What is Pure Functional Programming?

Pure Functional Programming is a programming style where functions behave predictably and do not depend on changing data.

A Pure Functional Program follows four important rules:

1. No State (Stateless Functions)
2. No Side Effects
3. Immutable Variables
4. Recursion Over Loops

---

# Rule 1: No State (Stateless Functions)

## What is State?

State refers to data that changes during program execution.

Example:

```java
private int counter = 0;
```

The value of `counter` can change over time.

---

## Bad Example (Stateful Function)

```java
public class StatefulExample
{
    private int counter = 0;

    public int increment()
    {
        return ++counter;
    }

    public static void main(String[] args)
    {
        StatefulExample obj =
                new StatefulExample();

        System.out.println(obj.increment());
        System.out.println(obj.increment());
    }
}
```

Output:

```text
1
2
```

---

## Why is this not Functional?

The method:

```java
increment()
```

depends on:

```java
counter
```

which changes every time the method executes.

The output depends on the current state.

This makes the function unpredictable.

---

## Good Example (Stateless Function)

```java
public class StatelessExample
{
    public static int increment(int value)
    {
        return value + 1;
    }

    public static void main(String[] args)
    {
        System.out.println(increment(0));
        System.out.println(increment(0));
    }
}
```

Output:

```text
1
1
```

---

## Why is this Functional?

The method:

```java
increment()
```

depends only on:

```java
value
```

which is provided as input.

There is no stored state.

Same input always produces the same output.

---

# Rule 2: No Side Effects

## What is a Side Effect?

A side effect occurs when a function changes something outside its own scope.

Examples:

* Modifying global variables
* Updating files
* Updating databases
* Sending emails
* Printing to console
* Changing shared objects

---

## Bad Example (Side Effect)

```java
public class SideEffectExample
{
    private static int total = 0;

    public static int addToTotal(int value)
    {
        total += value;
        return total;
    }

    public static void main(String[] args)
    {
        System.out.println(addToTotal(5));
        System.out.println(addToTotal(5));
    }
}
```

Output:

```text
5
10
```

---

## Why is this not Functional?

The method changes:

```java
total
```

which exists outside the method.

This is a side effect.

The same input:

```java
5
```

produces different outputs.

---

## Good Example (No Side Effects)

```java
public class NoSideEffectExample
{
    public static int add(int a, int b)
    {
        return a + b;
    }

    public static void main(String[] args)
    {
        System.out.println(add(5, 5));
        System.out.println(add(5, 5));
    }
}
```

Output:

```text
10
10
```

---

## Why is this Functional?

The method:

```java
add()
```

depends only on its parameters.

It does not modify external data.

No side effects occur.

---

# Rule 3: Immutable Variables

## What is Mutability?

A mutable variable can change after it is created.

Example:

```java
int sum = 10;

sum = 20;
```

The value changes.

This is mutable.

---

## Mutable Example

```java
int sum = 10;

sum = 20;
```

The variable changes state.

---

## Immutable Example

```java
final int sum = 10;
```

Now:

```java
sum = 20;
```

causes a compilation error.

The value cannot change.

---

# Why Does Functional Programming Prefer Immutability?

Because immutable values:

* Cannot be modified accidentally
* Reduce bugs
* Improve thread safety
* Make programs predictable

---

# Immutable Collections

Java provides immutable collections.

Example:

```java
List<String> names =
        List.of("John",
                "Mary",
                "David");
```

Attempting:

```java
names.add("Alex");
```

throws:

```text
UnsupportedOperationException
```

because the collection is immutable.

---

# Functional Operations on Immutable Collections

```java
List<String> names =
        List.of("John",
                "Mary",
                "David");

names.stream()
     .filter(name -> name.startsWith("J"))
     .forEach(System.out::println);
```

Output:

```text
John
```

The original collection remains unchanged.

---

# Rule 4: Recursion Over Loops

Pure Functional Programming avoids loops.

Instead, it uses recursion.

---

# Loop Example

```java
public static int factorial(int n)
{
    int result = 1;

    for(int i = 1; i <= n; i++)
    {
        result *= i;
    }

    return result;
}
```

Output:

```text
120
```

for:

```java
factorial(5)
```

---

## Why is this Less Functional?

The variable:

```java
result
```

changes repeatedly.

The loop counter:

```java
i
```

also changes.

State is being modified.

---

# Recursive Example

```java
public static int factorial(int n)
{
    if(n == 1)
    {
        return 1;
    }

    return n * factorial(n - 1);
}
```

Output:

```text
120
```

for:

```java
factorial(5)
```

---

# How Recursion Works

```text
factorial(5)
    =
5 * factorial(4)

factorial(4)
    =
4 * factorial(3)

factorial(3)
    =
3 * factorial(2)

factorial(2)
    =
2 * factorial(1)

factorial(1)
    =
1
```

Result:

```text
5 × 4 × 3 × 2 × 1
=
120
```

---

# Why Functional Programming Prefers Recursion

Recursion:

* Avoids loop counters
* Avoids mutable variables
* Produces cleaner logic
* Fits naturally with functional concepts

---

# Summary of the Four Rules

| Rule                 | Description                                     |
| -------------------- | ----------------------------------------------- |
| No State             | Functions should not rely on changing variables |
| No Side Effects      | Functions should not modify external data       |
| Immutable Variables  | Values should not change after creation         |
| Recursion Over Loops | Prefer recursive calls instead of loops         |

---

# Benefits of Following These Rules

## Predictable Behavior

Same input always produces the same output.

---

## Easier Testing

Functions can be tested independently.

---

## Easier Debugging

No hidden state changes.

---

## Better Thread Safety

Immutable data is naturally safer in multi-threaded applications.

---

## More Maintainable Code

Code becomes easier to understand and modify.

---

# Functional vs Imperative Programming

| Feature        | Imperative Programming | Functional Programming |
| -------------- | ---------------------- | ---------------------- |
| State          | Frequently Modified    | Avoided                |
| Side Effects   | Common                 | Avoided                |
| Variables      | Mutable                | Immutable              |
| Loops          | Preferred              | Recursion Preferred    |
| Predictability | Lower                  | Higher                 |
| Testing        | Harder                 | Easier                 |

---

# Summary

Pure Functional Programming follows four important rules:

1. No State
2. No Side Effects
3. Immutable Variables
4. Recursion Over Loops

By following these principles:

* Code becomes predictable.
* Functions become reusable.
* Testing becomes easier.
* Debugging becomes simpler.
* Multi-threaded applications become safer.

These principles form the foundation of modern Functional Programming and are heavily used with Java Streams, Lambda Expressions, Method References, and Functional Interfaces.
